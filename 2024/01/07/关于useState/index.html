React中的“反直觉”（1）：关于useState

关于React的争论在近日又引发了强烈讨论。有人说，React的新版文档写的很好，对React中易错的误区进行了详尽的阐释。也有人说，React的发展在不断加重开发者的心智负担。作为一个受众如此之广的框架，遇到这些相当“反直觉”的概念误区，没有去想办法做改进，反而让开发者去接受这些“反直觉”的概念，开发者是否对React太过宽容了？

但不论如何，对于React的开发者来说，这些“反直觉”的概念，为了避免出错，还是应当有一个准确的认知，这也是本人更新这个系列的初衷。文章不会深入地去解析原理，目的更多是帮助建立一个React希望我们建立的心智模型。

本系列的内容基于React18

那让我们从最常用的概念说起：useState。

setState 后发生了什么？

state 快照

React的官方文档将 state 比喻成一张快照，组件的每次重新渲染，会去使用拿到的最新的 state 快照，计算它的 props、事件处理函数和内部变量，从而去更新 UI。

简单来说，当 React 重新渲染一个组件时：

  1. React 会再次执行该函数组件
  2. 函数会根据当前渲染时的 state 快照返回新的 JSX 快照
  3. React 拿到新的 JSX 快照去进行 DOM 更新

setState 到底是同步还是异步？

先说结论：setState 不应该将被理解为异步，但它表现的很像“异步”。例如我们在同一个事件处理函数中，在更新了 state 后立即去取该 state 的值，会发现其还是更新前的值。

state 会带给我们这种似异步而又非异步的感觉，和 React 的调度机制密切相关。

在前文中我们已经了解了关于 state 快照的概念。事实上，React 只会使用触发交互时的快照进行调度。React 也会等到事件处理函数中的 所有代码都运行完毕再去处理 state 的更新。这样的机制使得整个更新过程，看上去很像异步。但这并非真正意义上的异步，只是React 会为了避免多次重复渲染而对 state 做批量更新，事实上仍然是在同一个call stack中进行的，只是改变了调用顺序而已。

这一机制使得一个 state 变量的值永远不会在一次渲染的内部发生变化，即使其事件处理函数的代码是异步的。

但在 React18 之前，置于异步函数中或者 js 原生事件中的 state 更新会是同步的。这是 React18 中的一个重大变化，也因此很多文章会称，在 React18 之后，所有的 state 更新都是“异步”的。

https://github.com/reactwg/react-18/discussions/21

综上所述，在 React18 之后，我们只需理解为，React 会使 state 的值始终”固定“在一次渲染的各个事件处理函数内部，即使在事件处理函数中更新了 state。
